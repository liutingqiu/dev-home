一个人的全栈运营团队

2026 年 6 月 1 日，我有了一个想法：「让 AI 全权运营一个小红书账号」。分析后发现手机端的物理限制无法突破，但换个方向——建一个内容矩阵，从数据采集到文章发布全程自动化——完全可行。

48 小时后，这个系统上线了。

系统全景

数据采集层: 7源爬虫 → 30分钟轮换 → 每日150+条原始数据
    ↓
内容生产层: 规则过滤 → 日报自动生成 + AI 深度文章
    ↓
分发层: dev-home 主站 + 知乎/掘金
    ↓
变现层: 搜索流量 → 个人品牌 → 接单转化

另外还有三个辅助系统：

  • Chat PWA：手机端的实时对话 + 灵感池
  • 邮件通知：异常告警 + 每日摘要
  • GitHub 备份：所有代码和内容云端存储

Chat PWA：出门也能联系 AI

这是整个系统里用户体验要求最高的部分。设计目标：

  1. PWA 全屏（可添加到手机主屏幕）
  2. 双模式：实时对话 + 灵感池
  3. 顶部状态栏显示 AI 当前任务
  4. 消息轮询 + 弹窗通知
  5. URL Token 鉴权

    6. 纯 HTML + CSS + JS 实现，零框架。Service Worker 做离线缓存，manifest.json 配置全屏模式。

    爬虫系统的设计考量

    最棘手的不是爬取逻辑——是反反爬：

    1. API 优先：GitHub、Hacker News、Reddit 有公开 API，不触碰 HTML
    2. 轮换间隔：同源 15 分钟一轮，避免触发限流
    3. User-Agent 轮换：3 个真实浏览器 UA 随机切换
    4. 优雅降级：爬失败的源不阻塞整体流程，标记错误后跳过

      5. 中文源（知乎、掘金、V2EX）的反爬更严格——知乎直接 403，掘金的 CSS 选择器一天一变。所以我把这些源标记为「实验性」，优先保证 GitHub + HN 的稳定。

      灵感池：不打断主流程的 idea 收集

      移动端最大的特点是「碎片化思考」。地铁上、排队时、睡前——想法一闪而过，打开笔记 App 打字太慢，等坐到电脑前已经忘了。

      灵感池解决了这个问题。手机上点 💡 → 输入想法 → 保存。AI 空闲时自动分析：

      • 关键词分类（AI / Web / 工具 / 安全...）
      • 可行性评估（高 / 中 / 低）
      • 技术方案草稿

      主人随时可以问：「我之前那个 XX 想法分析了吗？」AI 直接从存档调取。

      最意外的收获：邮件每日摘要

      每天早上 08:30，系统自动发邮件：

      爬虫状态: 2/2 源正常
日报发布: 5 期
灵感池: 3 条待分析
下次建议: 知乎近 3 天零互动，需调整标题

      这封邮件给了我一种「系统还活着」的安心感——不需要登录服务器，扫一眼邮箱就知道一切正常。

      复盘：48 小时做了什么

      | 阶段 | 内容 | 时间 |

      |---|---|---|

      | 需求分析 | 讨论可行性、漏洞、回环逻辑 | 4h |

      | 基础设施 | GitHub 仓库、目录结构、部署流水线 | 2h |

      | 通信系统 | Chat PWA + 5 个 API + Token 鉴权 | 4h |

      | 爬虫系统 | 7 源轮换 + cheerio + 错误重试 | 3h |

      | 日报系统 | 规则分类 + HTML 模板 + 原子部署 | 2h |

      | 博客系统 | MD → HTML 构建器 | 1h |

      | 辅助系统 | 邮件通知、灵感分析、计划任务 | 3h |

      | 内容生产 | 5 篇深度文章 | 3h |

      | 部署调试 | 阿里云部署、Chat 双向打通 | 6h |

      总代码量：约 3500 行（27 个新文件，1 个依赖）

      写在最后

      这个项目证明了：一个人的全栈运营团队，不是幻想。AI 不是替代你，是让你从重复劳动里解放出来——把时间花在真正需要创造力的地方。
