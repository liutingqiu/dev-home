# 远控源码技术评估报告

## 一、项目概况

| 项目 | 详情 |
|------|------|
| 原始项目 | Gh0st 远控改版（yuankong-master） |
| 代码规模 | 174 文件 / ~42,700 行 C++ |
| 开发环境 | Visual Studio 2015 + MFC |
| 架构 | C/S 模式：服务端 + 被控端 DLL |

---

## 二、代码现状问题

### 1. 安全层面 — 几乎为零

- **无身份认证**：代码中定义了 `TOKEN_AUTH = 100`（认证令牌），但在全部 45 个 `.cpp` 文件中**一行实现代码都没有**，纯属空壳定义。任何知道 IP 和端口的人都能直接连接控制。
- **通信无加密**：客户端与服务端之间的数据传输仅做了 LZ4/ZSTD 压缩，没有经过任何加密处理，网络抓包可直接还原控制指令和屏幕画面。
- **硬编码魔术字**：配置数据通过搜索 `0x1234567` 魔术字定位并覆写，极其容易被逆向和篡改。
- **明文配置**：IP、端口以明文写在 `remote.ini` 中，无任何保护。

### 2. 工程规范 — 学生作业级别

- **大量未完成的 TODO**：MFC 向导自动生成的注释（如 `// TODO: 在此添加控件通知处理程序代码`）超过 30 处原封不动保留，说明很多功能是半成品。
- **枚举定义重复维护**：客户端 `Common.h` 和服务端 `stdafx.h` 各自维护一份 80+ 值的相同枚举列表，修改一处极易遗漏另一处，属于典型的维护陷阱。
- **全局变量泛滥**：`g_2015RemoteDlg`、`m_iocpServer`、`g_ConnectAddress` 等全局状态散落各处，模块间强耦合，改动一处容易引发连锁崩溃。

### 3. 代码质量 — 存在已知设计缺陷

- **代码注释自曝问题**：`FileManagerDlg` 中注释写道 *"因为容易导致程序崩溃而出此策略"*，说明开发者已经发现了稳定性隐患，但采用了"加全局 vector 规避崩溃"的权宜之计而非从根本上修复。
- **错误处理粗糙**：多处使用 `catch(...)` 吞掉所有异常不处理；日志仅通过 `OutputDebugStringA` 输出，Release 版本完全静默，出问题无法排查。
- **测试代码混入项目**：`test.cpp` 中包含拼写不一致的函数（`DebugPrivilege` / `DebugPrivilege`），测试环境和生产代码未分离。

### 4. 技术栈陈旧

- 基于 MFC（微软 1992 年发布的框架），强依赖 Windows 平台，无法跨平台。
- 全程裸指针 + `new/delete` 手动管理内存，无 C++11/14/17 的智能指针和 RAII 机制。
- 无版本控制规范（`.gitignore` 缺失，`.user` 本地配置文件提交到了仓库）。

## 三、结论

| 维度 | 当前水准 | 商用要求 | 差距 |
|------|:--------:|:--------:|:----:|
| 功能完整度 | 60% | 95%+ | 大 |
| 代码规范 | D+ | B+ 以上 | 很大 |
| 安全性 | 0 | 必须 | 极大 |
| 可维护性 | 差 | 良好 | 大 |

当前源码是一个**爱好者级别的 Gh0st 改版**，能跑通基本功能但远未达到商用交付标准。如果需要在现有基础上进行功能修改（如去掉密码/口令），**首先需要投入大量精力做代码治理和安全加固**，否则改动风险极高。

---

## 四、后续建议

若客户希望基于此源码进行定制开发，建议的工作顺序：

1. **代码清理**：统一枚举定义、清理全局变量、移除死代码和空 TODO
2. **安全加固**：实现认证机制、通信加密、防逆向保护
3. **架构重构**：模块解耦、引入智能指针、统一错误处理
4. **功能修改**：在健康代码基础上进行客户要求的定制

以上评估基于对全部 174 个文件、~42,700 行源码的逐项审查。
